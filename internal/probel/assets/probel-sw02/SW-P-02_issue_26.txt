1. 
INTRODUCTION
This protocol provides a robust and asynchronous method of controlling routers and is the preferred method of communication.  New message commands are added as required.
2. ELECTRICAL
Normal electrical parameters are:
		RS485/RS422, 8 Bit DATA, 1 STOP Bit, EVEN Parity, 38.4K Baud,
But the protocol will operate with any common RS485/RS422 and RS232 data parameters.
3. PROTOCOL
3.1. Message Format
All messages conform to the following format:
SOMCOMMANDMessageChecksum
where
		SOM		-	Start of message = FF hex
		COMMAND	-	Command byte
						(this also defined message length)
						All command bytes listed are in decimal
		Message		-	May be zero or more bytes
		Checksum	-	7 bit, 2's complement sum of command and message field 
						(m.s.b. = 0)

3.2. Message Commands
All command numbers referred to in this document are in decimal.
The following commands are defined:
3.2.1. DATABASE CHECKSUM REQUEST Message
This message is used to request a database checksum.
Command Byte:	00
Message:	Byte 1 -	00 = request a checksum


3.2.2. DATABASE CHECKSUM RESPONSE Message
This message responds with the checksum of a database after a DATABASE CHECKSUM REQUEST has been received.
Command Byte:	00
Message:	Byte 1 -	bits [0-3]	:	0001 = checksum response
					bits [5]	:	high bit of low byte of checksum
					bits [6]	:	high bit of high byte of checksum
Byte 2 -	bits [0-6]	:	bits [0-6] of checksum (low byte)
Byte 3 -	bits [0-6]	:	bits [0-6] of checksum (high byte)

3.2.3. INTERROGATE Message
Request for tally information.  A device responds to this message with a TALLY message 3.2.5.
Command Byte:	01
Message:	Byte 1 -	Multiplier
					(this field allows sources and dests of up to 1023 to be used
					and provides source status info from TDM and HD Digital
					Video routers).
					bit 7		:	0
					bit 4-6	:	Dest number DIV 128
								Note:	When all bits set to 1, the
										subsequent 7 bit byte refers to
										a Monitoring row,  hence
										practically, this means Dests
										0-895 are main	router dests &
										dests 896 to 1023 are monitor
										rows.
					bit 3		:	either
								TDM and Digital Video "bad" source on
								output status
								(1 = bad source)
								or
								Crosspoint Update Disabled.
								(1 = crosspoint updated disabled.)
								See Section 3.2.13 for details
								or
								not used.
					bit 0-2	:	Source number DIV 128
								(Not applicable to this command)
								(always 0 for this command).
 Byte 2 -	Destination number MOD 128
 
 
 
3.2.4. CONNECT Message
This message requests a route to be made through the router.  A device will make the route and respond with a CONNECTED message (3.2.6) that also performs an effective acknowledgement.
Command Byte:	02
Message:	Byte 1 -	Multiplier as defined in (3.2.3)
     Byte 2 -	Destination number MOD 128
     Byte 3 -	Source number MOD 128
Note:	A single CONNECT command will suffice to set two crosspoints in the TDM 4-Wire Router 2510 local controller. The 2510 local controller will confirm this by sending two CONNECTED messages.

3.2.5. TALLY Message
This message returns tally information in response to an interrogating device.  It also performs an effective acknowledgement to the INTERROGATE message.
Command Byte:	03
Message:	Byte 1 -	Multiplier as defined in (3.2.3)
     Byte 2 -	Destination number MOD 128
     Byte 3 -	Source number MOD 128
Note:	Source number 1023 is reserved to indicate destination out of range.

3.2.6. CONNECTED Message
This message is issued by a device after it has made a router through a router, usually in response to a CONNECT request.  It is issued on ALL ports of the interface device.
Command Byte:	04
Message:	Byte 1 -	Multiplier as defined in (3.2.3)
     Byte 2 -	Destination number MOD 128
     Byte 3 -	Source number MOD 128
     
     
     
     
     
3.2.7. CONNECT ON Go Message
Used in Salvo switching situations.  Routing information is held in the receiving device until activated by GO command (3.2.8).  A device will respond with a CONNECT ON GO ACKNOWLEDGE message (3.2.14) to indicate that the routing information has been stored successfully.
Command Byte:	05
Message:	Byte 1 -	Multiplier as defined in (3.2.3)
     Byte 2 -	Destination number MOD 128
     Byte 3 -	Source number MOD 128
     
3.2.8. GO Message
Triggers the receiving device to set all routes in/clear the previously received CONNECT ON GO messages.  A GO DONE acknowledge message (3.2.15), will be issued to indicate that the command has been executed.
Note:	No individual CONNECTED messages (3.2.6) are issued.  It is the
		responsibility of the controlling/listening devices to poll for the information,
		i.e. INTERROGATE commands (3.2.3).
Command Byte:	06
Message:	Byte 1 -	00 Set previously received messages
					01 Clear previously received messages

3.2.9. STATUS REQUEST Message
Triggers the receiving device to respond with a status response message (3.2.10, 3.2.11, 3.2.12, 3.2.18 or 3.2.19).
Command Byte:	07
Message:	Byte 1 -	Value 0 requests response by LH controller
					Value 1 requests response by RH controller
					(single controller systems or TDM 4-Wire Router value = 0)
3.2.10. 
STATUS RESPONSE - 1 Message
Issued by a 2513/2514/2515/2659/2440/2441/2442/2443 card or a TDM 4-Wire router (2510) card in response to the STATUS REQUEST message (3.2.9). This message indicates the number of faulty 8 x 8 crosspoint cards or faulty 4-Wire router I/O modules or faulty crosspoint modules on TM24 (2659) or Freeway (2440/1) or Nebula (2442/3) in the router unit.
Command Byte:	08
Message:	Byte 1 -	Number of faulty blocks/modules
     Byte 2 -	Faulty 2538 number (0-15)
			or 4-Wire router frame number (0-7)
			or level number on TM24 (0-3)
			or level number on Freeway/Nebula (0-7)
     Byte 3 -	Faulty crosspoint card number (0-15),
			or 4-Wire router I/O module number (0-16)
			(Module 16 = Idle 2510 controller card)
			or TM24 crosspoint module number (0-31)
			(0-7 = main audio, 16-23 = expansion audio,
			8-15 = main video, 24-31 = expansion video),
			or Freeway/Nebula crosspoint module number (0-31)
Bytes 2 and 3 are repeated as many times as byte 1 indicates that there are faulty blocks.

3.2.11. STATUS RESPONSE - 2 Message
Issued by a 5007/5107 TDM controller card in response to the STATUS REQUEST message (3.2.9).
Command Byte:	09
Message:	Byte 1 -	bit 7	:	0
					bit 6	:	0 = Active systems
							1 = Idle system
					bit 5	:	0 = No bus fault
							1 = Bus fault
					bit 4	:	0 = No overheat (system is cool!)
							1 = Overheat
					bit 0-3 :	Always 0






3.2.12. STATUS RESPONSE - 3 Message
Issued by an HD local control card (2412/2413/2414) in response to the STATUS REQUEST message (3.2.9).  This message indicates the number of faulty input/crosspoint/output card in the switcher unit.
Command Byte:	10
Message:	Byte 1 -	Number of faulty modules
     Byte 2 -	Faulty module number (1-58)
     Byte 3 -	Faulty module type												(0 = input card, 1 = crosspoint card, 2 = output card)
Bytes 2 and 3 are repeated as many times as byte 1 indicates that there are faulty modules.

3.2.13. ENABLE CROSSPOINT UPDATE Message
Issued to a router to enable actual crosspoint setting to take place, otherwise the crosspoint card will store requests until enabled.
Note:	
Currently only implemented in the HD local control card and this feature is jumper selectable.  Also this command has no effect on HD Digital video/RS422 routers or MADI routers.
When this mode is enabled, bit 3 of the multiplier byte (see Section 3.2.3) is used to indicate the crosspoint update status.
Command Byte:	11
Message:	No message field.

3.2.14. CONNECT ON GO ACKNOWLEDGE Message
This message is issued by a device in response to a CONNECT ON GO request (3.2.7).
Command Byte:	12
Message:	Byte 1 -	Multiplier as defined in (3.2.3)
				(Except bad source bit always 0)
     Byte 2 -	Destination number MOD 128
     Byte 3 -	Source number MOD 128.
     
     
3.2.15. GO DONE ACKNOWLEDGE Message
This message is issued by a device in response to a GO message, (3.2.8).  If crosspoints are set then the message is issued on all ports of the receiving device.
Command Byte:	13
Message:	Byte 1 -	00 = Crosspoints set
					01 =	Stored crosspoints cleared
					02 = No crosspoints to set/clear

3.2.16. SOURCE LOCK STATUS REQUEST Message
Triggers the receiving device to respond with a SOURCE LOCK STATUS response message (3.2.17).  This command is only applicable to the HD digital video routers.
Command Byte:	14
Message:	Byte 1 -	Value 0 requests response by LH controller
					Value 1 requests response by RH controller
					(single controller systems, value = 0)

3.2.17. SOURCE LOCK STATUS RESPONSE Message
Issued by an HD series local control card in response to the Source Lock Status request message (3.2.16).  This message indicates the source lock status on the input modules.
Command Byte:	15
Message:	Byte 1 -	Bytes in message DIV 128
     Byte 2 -	Bytes in message MOD 128
     Byte 3 -	bit 0 : source 0 lock state
			bit 1 : source 1 lock state
			bit 2 : source 2 lock state
			bit 3 : source 3 lock state
			bits 4-7 : 0
     Byte 4 - 	Byte (max source DIV 4)
			as byte 3 for each group of 4 sources.
Notes:		1.	Source lock ok value = 1
			2.	If an input card missing/not installed then source lock value for
				the sources on that module will be returned as 0.




3.2.18. STATUS RESPONSE - 4 Message
This is sent in response to a STATUS REQUEST message (3.2.9).  It indicates which modules in an XD router are faulty.
Command Byte:	16
Message:	Byte 1 -	Number of faulty modules
     Byte 2 -	1st faulty card number DIV 128
     Byte 3 -	1st faulty card number MOD 128
     Byte 4 -	2nd faulty card number DIV 128
     Byte 5 -	2nd faulty card number MOD 128
	..
	..
     Byte n -	nth faulty card number DIV 128
     Byte n+1-	nth faulty card number MOD 128
The maximum number of faulty cards reported is limited to 127.
The total number of message bytes is equal to:
     (No. of faulty cards x 2) + 2
     
3.2.19. STATUS RESPONSE - 5 Message
This is sent in response to a STATUS REQUEST message (3.2.9).  It indicates which modules in a router made up of multiple routers are faulty. This message would normally be implemented on an interface unit that combines several router frames to make a logically larger router.
Command Byte:	17
Message:	Byte 1 -	Number of Faulty modules DIV 128
			Byte 2 -	Number of Faulty modules MOD 128
			Bytes 3-6	1st Fault.
					The format of the returned fault is dependent on the
					router type. Sections 3.2.19.1  - 3.2.19.4  define the formats
					available.
Bytes 3 to 6 are repeated as many times as bytes 1 and 2 indicate there are faulty modules.





3.2.19.1  Type 1 Routers
This type of router covers all router controller types defined in the Status Response-1 message (See 3.2.10 for details).
Each returned fault has the following message structure:
     Byte 1 	-	Frame/Port no (00-127)
     Byte 2 	-	Fault Type
				01H =	Primary Fault - Port failure
						(comms failure with this section of router)
				11H = 	Secondary Fault - Module failure 
				21H =	Secondary Fault - PSU/Fan failure
				31H =	Secondary Fault - Controller failure
Byte 3 & 4 -	Both 00H if Primary Fault	
			or
			as bytes 2 & 3 of Status Response-1 Message if 
			Secondary Fault - Module failure
			or 
 			if Secondary Fault - PSU/Fan failure
 			Byte 2 = 	PSU Fails 
 					(bit 7 = 0, bits 0 - 6 = individual PSU fails,
 					bit = 1 = failed),
 			Byte 3 =	Fan Fails
 					(bit 7 = 0, bits 0 - 6 = individual fails,
 					bit = 1 = failed)
			or 
 			if Secondary Fault - Controller failure
 			Byte 2 = 	Controller Fails 
 					(bit 7 = 0, bits 0 - 6 = individual Controller fails,
 					bit = 1 = failed),
					(normally only bits 0 and 1 used as only a
					maximum of two controllers).				
 			Byte 3 =	00H

3.2.19.2 Type 2 Routers
This type of router covers all controller types defined in the Status Response-2 message (See 3.2.11 for details).
Each returned fault has the following message structure:
     Byte 1	-	Frame/Port no (00-127)
     Byte 2	-	Fault Type
				02H =	Primary Fault - Port failure
						(comms failure with this section of router)
				12H =	Secondary Fault - Module failure
Byte 3	-	00H if Primary Fault
			or
			as byte 1 of Status Response-2 Message if secondary
			fault
     Byte 4	-	Always 00H
     
3.2.19.3  Type 3 Routers
This type of router covers all controller types defined in the Status Response-3 message (See 3.2.12 for details)
Each returned fault has the following message structure:
     Byte 1	-	Frame/Port no (00-127)
     Byte 2	-	Fault Type
				03H =	Primary Fault - Port failure
						(comms failure with this section of router)
				13H = 	Secondary Fault - Module failure
     Byte 3 & 4 -	Both 00H if Primary Fault	
				or
				as bytes 2 & 3 of Status Response-3 Message if 
				secondary fault
     
3.2.19.4  Type 4 Routers
This type of router covers all controller types defined in the Status Response-4 message (See 3.2.18 for details)
Each returned fault has the following message structure:
     Byte 1	-	Frame/Port no (00-127)
     Byte 2	-	Fault Type
				04H =	Primary Fault - Port failure
						(comms failure with this section of router)
				14H = 	Secondary Fault - Module failure
     Byte 3 & 4	Both 00H if Primary Fault	
				or
				as bytes 2 & 3 of Status Response-4 Message if 
				secondary fault
     
3.2.20. STATUS REQUEST 2 Message
Triggers the receiving device to respond with a status response message 6 (3.2.21).
Command Byte:	18
Message:	Byte 1 -	Value 0 requests response by controller A
					Value 1 requests response by controller B
					(single controller systems  = 0)



3.2.21. STATUS RESPONSE - 6 Message
Issued by a controller card in response to the STATUS REQUEST 2 message (3.2.20). This message indicates that there are one or more types of these faults in the system. To find out more information about the faults use the command 07 (3.2.9) to request this. This message is broadcast on all ports whenever the fault status changes.
Command Byte:	19
Message:	Byte 1 -	bit 7	:	0
					bit 4-6:	0 (reserved for future use)
					bit 3	:	1 = Controller Fault
					bit 2	:	1 = Module Fault (Crosspoint/Input/Output)
					bit 1	:	1 = Fan Fault 
					bit 0 :	1 = PSU Fault
			Byte 2 -	bit 7	:	0 
					bit 0-6:	0 (reserved for future use)

3.2.22. MIXER INTERROGATE Message
The TDM 4-Wire Router has potential of mixing up to 32 sources with different gain coefficients.  The MIXER INTERROGATE message is sent by the control system to inquire about all sources contributing to the given output.  The TDM 4-Wire Router controller (2510) will reply with MIXER TALLY DUMP (3.2.24).
Command Byte:	21
Message:	Byte 1 -	Multiplier
					bits 0-3	:	All 0
					bits 4-6	:	Destination number DIV 128
					bit 7		:	0
     Byte 2 -	Destination number MOD 128
     
     
     
     
     
     
     
     
     
     
3.2.23. MIXER CONNECT Message
This message is sent by the control system to the TDM 4-Wire Router to set a mixing crosspoint with the given gain coefficient.  The command will fail if there are already 32 sources contributing to the given output.  To cancel a selection made previously, a MIXER CONNECT message with gain 0 has to be sent.
Command Byte:	22
Message:	Byte 1 -	Multiplier
					bits 0-2	:	Source number DIV 128
					bit 3		:	0
					bits 4-6	:	Destination number DIV 128
					bit 7		:	0
   Byte 2 -	Destination number MOD 128
    Byte 3 -	Source number MOD 128
 Byte 4 -	(gain x 8) DIV 128
   Byte 5 -	(gain x 8) MOD 128
Notes:		1.	Gain coefficients are given in the linear scale 0-40 in steps of
				0.125.
			2.	MIXER CONNECT will set only the forward router, unlike the
				CONNECT command which is used to make bi-directional
				routes.

3.2.24. MIXER TALLY DUMP Message
This message is sent by a TDM 4-Wire Router in response to a MIXER INTERROGATE message (3.2.22). It returns information about all sources contributing to the given output.
Command Byte:	23
Message:	Byte 1 -	Destination number DIV 128
     Byte 2 -	Destination number MOD 128
     Byte 3 -	Number of sources routed with non-zero gain.
     Byte 4 -	Input/Gain Multiplier	
			bits 0-2	:	Source number DIV 128
			bit 3		:	0
			bits 4-6	:	(gain x 8) DIV 128
			bit 7		:	0
     Byte 5 -	Source number MOD 128
     Byte 6 -	(gain x 8) MOD 128
Bytes 4-6 will be repeated as many times as byte 3 indicates (maximum of 32 times).

3.2.58. ROUTER CONFIGURATION RESPONSE - 1 Message
This is sent in response to a ROUTER CONFIGURATION REQUEST message (3.2.57). It contains a bit map showing which levels are configured and the number of destinations and sources on each level. When controlling a router with multiple levels the destinations are stacked on top of each other. E.g. For a router consisting of 3 levels of 128x128, 64x64 and 32x32, the levels would be controlled as follows. Level 1 crosspoints using destinations and sources in the range 0 - 127, level 2 using destinations in the range 128 - 191, sources in the range 0 - 63 and level 3 using destinations in the range 192 - 223, sources in the range 0 - 31. The size of this message depends on the number of level in the router. For applications that do not need the level information but need the maximum source and destination numbers, the number of destinations can be calculated by adding the size of each of the levels together and the number of sources by taking the maximum on any level.
Command Byte:	76
Message:	Byte 1 -	Levels bit map (Bits 0 - 6 = levels 21 - 27, Bit 7 = 0).
     Byte 2 -	Levels bit map (Bits 0 - 6 = levels 14 - 20, Bit 7 = 0).

     Byte 4 -	Levels bit map (Bits 0 - 6 = levels 0 - 6, Bit 7 = 0).
Bytes 1 - 4 provide a 28 bit, bit map. Bit 0 represents the first level. The bit is set to indicate the level is available on the router and that a destination and source size is returned in the message.
     Byte 5 -	First level, No of Destinations Multiplier

     Byte 6 -	First level, No of Destinations MOD 128

			Bits 0 - 6	:	Source number DIV 128
			Bit 7		:	Always 0

3.3. Notes on Implementation
3.3.1. Philosophy
The philosophy of this message transfer protocol is that a controller assumes that a command has been successful until proved otherwise through, for example, a timeout.
In Pro-Bel controllers, a switcher is continually INTERROGATED as a background function to ensure that tally information is up-to-date.  In System1/System2 when a CONNECT message is issued, the control system assumes that the connection will be successful and updates its internal tables accordingly.  After a CONNECT message is issued, TALLY messages are ignored for a preset time interval of approx. 100mS.  This ensures that any outstanding TALLY messages in the controlled device does not overwrite the new CONNECT information in the controller tally table.  In System3/Aurora the internal tally table is not updated until the connected message is received.
The background INTERROGATE function performed by a controller also indicates data link failure by timeouts on the TALLY message in the following manner :
When an INTERROGATE message is issued, a timeout counter is initialised.  If a TALLY message is not received before the counter reaches zero, then a data link failure is adjudged to have occurred.
Further Note, therefore, that the timeout period must be less than the INTERROGATE interval and that this implies a fundamental lower limit to the INTERROGATE poll time of
INTERROGATE send time
•	peripheral service time
•	TALLY + response time
•	TALLY service
Theoretically this could be as fast as
		1.25 + 1 + 1.25 + 1/2 = 4 mS.
Practically this should be no greater than an average of 1 INTERROGATE per 20 mS.

3.3.2. Further Criteria
Further criteria to be met by any implementation of this protocol are :
i)		A system must successfully service messages being received 					simultaneously on all ports.
ii)	A system must successfully service up to 20 consecutive CONNECT 				messages on any port, at any one time.
iii)	A system must successfully service an average of 1 CONNECT message 			every 10 mS.
