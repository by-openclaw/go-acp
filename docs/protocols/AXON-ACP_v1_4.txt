Introduction

This document describes the two protocols first introduced in the Synapse
product range. The protocol described in part one is the Axon Control Protocol
(ACP) and the second part describes the AxonNet object access protocol.

Axon Control Protocol

The ACP is a wrapper protocol for the UDP or TCP transport protocol. This thin
layer was added to make multiple cards accessible using one IP address and port
number. The ACP layer gives transparent access to the AxonNet objects.

The procedure for finding ACP servers on a network is described in the SDK.
Only a preliminary version is available now.

AxonNet objects

AxonNet objects are the structure of management information. This part describes
how objects can be accessed by using methods and defines the binary structure
for these objects. Each Synapse card holds a number of objects. Objects are
divided in object groups and each object has an object ID number. Note the
object ID numbers between card software revisions are subject to change.

Methods are executed using an ACP transaction from a client on a remote server.
The sequence of a client invoking a method (request) and a card returning a value
(reply) is called a transaction. Changes on objects can be followed by listening to
ACP announcements. These announcements can be generated by an ACP server
and are sent to all clients connected to the network.

Alarm objects can generate a spontaneous Alarm announcement or `Event'. These
Alarms are used to inform ACP clients about a changing condition on a card.

PART 1 - ACP

This chapter describes the Axon Control Protocol (ACP). ACP is designed for
control and monitoring applications for the Axon Synapse system. ACP uses IP for
its network layer and UDP for its transport layer. The physical layer is 10Base-T or
1000Base-T Ethernet.

The following diagram shows the route (thick lines) that ACP packets travel
through the TCP/IP Protocol Suite.

UDP is a connectionless protocol that does not guarantee packets delivery. To
make UDP work in a reliable way, ACP adds transaction ID numbers and a
timeout and retransmission mechanism. Flow control is achieved by serializing the
transactions.

ACP adds a message subaddress for addressing multiple cards using one IP
address. The message type field is added for discriminating between incoming and
outgoing transactions.

IP / UDP issues

Because the ACP payload is relatively small (<= 134 bytes), IP fragmentation and
IP options is not supported in ACP.
The maximum size of the IP datagram is 169 bytes (20 bytes IP Header + 8 bytes
UDP Header + 7 bytes ACP Header + 134 bytes ACP data).
The UDP checksum must be implemented because this is the only test for ACP
data integrity.

ACP Message Types

ACP defines two classes of messages for communication:

    1. Broadcast Messages:

         Servers as well as clients can initiate Broadcast Messages. Broadcast
         Messages are Subnet Broadcasts and are only received by the nodes
         within a subnet. Broadcast Messages are never acknowledged thus data
         delivery is not guaranteed.

         A Broadcast initiated by a server is called an Announcement.
         Announcements are generated on a server to inform all clients on a subnet
         of changes on control values or status items.

         If obtaining status is critical, transaction messages must be used instead
         (polling).

         The only broadcast message clients are allowed to invoke a Broadcast
         getValue(FrameStatus) Method. This will trigger all rack controllers on
         the subnet to reply with its FrameStatus value to that client.

    2. Transaction Messages:

         These are unicast messages sent between controllers (clients) and racks
         (servers) in order to perform transactions such as changing parameters or
         requesting status information. Two basic messages are defined: Request
         Messages and Reply Messages.

         If a client wants to perform some action on a server, it must first send a
         Request Message to that server. A server will respond by sending a
         Reply back to the client. Optionally, the server may announce (broadcast)
         the Reply on the subnet or additionally direct the Reply to a Gateway.

         If, for some reason, an error occurred during the handling of a Request
         Message, a server will return a special Reply Message, called a Reply Error
         Message, which contains an error description code.

Timeouts

Applications using ACP should implement a timeout mechanism. The mechanism
should operate on the client side and only on transmitted request messages. It is the
responsibility of the application developer to implement a solid timeout
mechanism. A possible mechanism is described below:

If a client transmits a Request Message (starts a transaction), it starts a timeout
counter. If the transmitted Request Message or Reply Message gets lost, no reply
will arrive from the server and the timeout counter will timeout.

ACP Announcements

When multiple clients are connected to a Synapse network, it is important that a
change made by one client to a card is announced to all other clients on the
network. Announcements are generated in the following situations:
1. A change occurs in a frame:

                             13
ACP ­ AXON CONTROL PROTOCOL                           REVISION 1.4

         · A card is inserted

         · A card is removed

         · A card is powering up

         · A card is in an error state

         · A card is in Boot Mode

2. A change occurs in a card's control item:

         A client has changed a control parameter. The new value of the control
         parameter is "announced" to all other clients. The source of this
         announcement is the rack controller. The controller listens to the replies of
         setValue, setDefault, setInc and setDec methods and announces it on the
         network.

3. A change occurs in a card status item:

         The status of some property on a card has changed. The new status value
         is "announced" to all clients. The source of this announcement is the card.

4. An Alarm is generated on a card:

         An Alarm event is generated on a card or frame. This event is
         "announced" to all clients. The information transmitted by such a message
         is volatile. It cannot be retrieved from the system after transmission.

Alarm Announcements can be individually enabled by setting their priority value to
a value greater than zero. Both Status and Alarm Announcements can be globally
disabled on the card by setting the first Alarm object (called `Announcements') to
zero.

The broadcasting of announcements can be enabled or disabled on the Synapse
rack controller. Because Announcements are only broadcasted within the sub-net,
the rack controller also supports forwarding Announcements to one IP Address
(which may be on a different sub-net).

PART 2 ­ AxonNet objects

An Object is an entity on a Card that has a value. Examples are Video Gain or
Input Status. Depending on the access rights of an Object, the value can be read,
written, incremented, decremented or set to default.

Objects are divided into Objects Groups. Each Object Group has a Group ID and
each Object in a Group has an Object ID. The Group ID and the Object ID are
byte sized indexes. Objects are not guaranteed to remain at the same index location
for different card firmware revisions. AxonNet client applications must
implement either a runtime discovery of object IDs or have runtime user
configurable object IDs.

A number of Properties are common to several Objects:

· A Label String holds an Item description (Video Gain, H-Phase, Format....).

· A Unit String holds a value postfix description (uS, secs, pixs, dB...). The

· The Default Value is fixed value, defined by Axon. It can be used to reset a
    value.

· The Minimum Value and Maximum Value properties are the limits of a
    value. When exceeding a limit, the Card will return either the Maximum Value
    (in case of a positive number) or the Minimum Value (in case of a negative
    number).

· The Default Step Size denotes the increment value for an Object numeric
    value. Most times the Default Step Size will be 1.

· A Byte is an unsigned 8-bit value.

· An Integer is a 16-bit signed value, two's complement.

Appendix: Recommended Practices

        This section of the document describes the recommended practices when writing a
        software driver for ACP.

      ACP Driver Forward and Backward Compatibility

        Each ACP controlled product has a defined number of objects in the control,
        status and alarm groups. These objects are defined for a specific software version
        of each product, details of which can be issued from Axon's support department.

        As the development cycle of these products continue features are added. This, in
        turn, leads to new objects being potentially added to one or more of these groups.

        These new objects will be added in the most logical position within the group, thus
        leading to the potential issue of the same object changing its object ID over
        different software revisions, as mentioned previously in the document.

        For this reason in order for software drivers to maintain both forward and
        backward compatibility with ACP controlled products it is recommended that the
        software uses the objects Label String rather than its object ID to identify the
        object to be accessed.

        The Label String will ALWAYS be unique within the products object group to
        which it belongs. Using this field as the unique identifier will give the benefit of
        being able to control/monitor different revisions of the same product at the same
        time and also similar types of product (provided that the Label String fields
        remain the same).

        As ACP is driven using the object Group and object ID, it is recommended that
        on initiating control of a specific instance of an ACP product all objects for that
        group are read using the getObject method so that an internal mapping table can
        be formed to correlate the objects Label String to its object ID. From this point
        on the client software should ideally refer to an Object by its Label String and
        only when sending/receiving a message to/from the device referring to its object
        ID.
