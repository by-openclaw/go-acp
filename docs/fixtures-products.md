# Product fixture library — layout and rules

Per-product, per-firmware captures that let any dev replay a device offline and see what changed between firmware versions. Generated by tooling; not hand-edited.

---

## Path shape

```
tests/fixtures/products/
├── README.md
└── <manufacturer>/                     axon | lawo | dhd | by-systems | ...
    └── <product>/                      DDB08 | CDV08v06 | mc56 | xs-core | ...
        └── <protocol>/                 acp1 | acp2 | emberplus
            ├── CHANGELOG.md            generated on each new version add
            └── <version>/              v2.3 | 9092 | firmware-A.B.C
                ├── meta.json           identity, fingerprint, source tool
                ├── wire.jsonl          raw frames (replay source)
                └── tree.json           canonical export (replay destination)
```

A product folder can contain multiple protocol subfolders — one physical card sometimes exposes several interfaces (e.g. an Axon DDB08 speaking both ACP2 and Ember+). Each protocol has its own version lineage and its own `CHANGELOG.md`.

### Slug rules

| Segment | Case | Rule |
|---|---|---|
| `<manufacturer>` | lowercase, hyphen for spaces | Vendor slug. `axon`, `lawo`, `dhd`, `by-systems`, `calrec`. |
| `<product>` | preserve vendor casing | Product identifier as the vendor writes it. `DDB08`, `CDV08v06`, `mc56`, `xs-core`. No spaces — replace with hyphens. |
| `<protocol>` | lowercase | Exactly one of `acp1`, `acp2`, `emberplus`. |
| `<version>` | as reported by device | ACP1/ACP2: firmware version from identity block (`v2.3`, `2.3.1`). Ember+: provider port or named release (`9092`, `lawo-v7.12`). Use what the device itself returns so lookup is deterministic. |

## `meta.json` schema

Locked schema. Extra keys allowed but not required.

```json
{
  "protocol": "acp2",
  "manufacturer": "Axon",
  "product": "DDB08",
  "version": "2.3",
  "version_kind": "firmware",
  "discovered_at": "2026-04-19T13:20:00Z",
  "description": "ACP2 Audio Processor card",
  "dm_fingerprint": "sha256:abc123…",
  "object_count": 214,
  "capture_tool": {
    "name": "acp",
    "version": "0.2.0",
    "git_tag": "v0.2.0",
    "git_commit": "7db5149"
  },
  "notes": ""
}
```

| Field | Required | Meaning |
|---|---|---|
| `protocol` | yes | One of `acp1`, `acp2`, `emberplus` — matches the directory segment. |
| `manufacturer` | yes | Vendor name, preserving vendor casing (`Axon`, `Lawo`, `DHD`). Directory segment uses the lowercase slug form. |
| `product` | yes | Product identifier — matches the directory segment. |
| `version` | yes | Version string — matches the directory segment. |
| `version_kind` | yes | `firmware` for hardware devices, `software` for Ember+/soft providers, `release` for named releases. |
| `discovered_at` | yes | UTC ISO-8601 with second precision. Timestamp of the capture run, not a DM property — included so fixtures can be audited. |
| `description` | no | Free text from the device's identity block (vendor-written, same across every instance of this firmware). |
| `dm_fingerprint` | yes | `sha256:` prefix + hex digest over canonical `tree.json` content — the DM's content-address. Two captures of the same firmware produce identical fingerprints across different physical devices. |
| `object_count` | yes | Number of leaves in `tree.json`. |
| `capture_tool` | yes | Structured object identifying the binary that produced the capture. See below. |
| `notes` | no | Free text for the engineer who captured (test context, what was pinned, known issues). |

### Deliberately excluded from meta.json

**Instance-level facts are not DM-intrinsic** and do not belong here. Two different physical DDB08 v2.3 cards produce identical DMs regardless of where they're deployed or which unit they are. The following were considered and excluded:

| Field | Why excluded |
|---|---|
| `slot` | Deployment detail (which chassis slot holds the card). Same card DM in slot 0 or slot 3. |
| `device_ip` | Network deployment detail. Same card DM at 10.41.40.195 or 192.168.1.50. |
| `serial_number` | Unit-level identity. Two serialised DDB08 v2.3 cards on the rack share the DM but have different serials. |
| `mac_address` | Same rationale as `device_ip`. |
| `rack_id`, `facility_id`, etc. | Topology facts — belong in the Catalog (see `docs/VISION.md` §7), not here. |

Instance-level facts live in the **Catalog** at a higher layer. The DM fixture library answers *"what does this firmware version look like?"* — the Catalog answers *"where is each instance of this firmware deployed?"*.

### `capture_tool` object

Four fields — all required once `acp extract` (#36.b) ships. Stamped automatically by the extractor from `-ldflags`-embedded build info.

| Field | Meaning | Example |
|---|---|---|
| `name` | Which binary produced the capture | `"acp"` |
| `version` | Release version (semver, matches the git tag without `v` prefix) | `"0.2.0"` |
| `git_tag` | Git tag at the exact build commit — `vX.Y.Z` on a tagged release, `<nearest-tag>-<N>-g<sha>` otherwise (output of `git describe --tags`) | `"v0.2.0"` |
| `git_commit` | Short SHA (7 chars, output of `git rev-parse --short HEAD`) | `"7db5149"` |

**Why every field matters:**
- `version` — human-readable reference.
- `git_tag` + `git_commit` — exact source provenance. If a regression shows up later in a replay test, we can `git checkout <git_commit>` and reproduce the exact capture-time behaviour. Especially important for unreleased builds (`git_tag` will show `vX.Y.Z-N-gSHA` for those).

If a capture is produced from an untagged, dirty worktree, `git_tag` carries the `-dirty` suffix (e.g. `"v0.2.0-3-g7db5149-dirty"`). Such captures should not be committed — fix the worktree first.

### Fingerprint rule

`dm_fingerprint` is the SHA-256 of `tree.json` rendered with canonical JSON encoding (sorted keys, UTF-8, LF line endings, no trailing newline). Recompute with `acp extract` or `openssl dgst -sha256 tree.json`.

Identical firmware → identical fingerprint across captures. Different fingerprint on the same stated version = the device wasn't what the label claimed, or the canonical encoder moved underneath. Audit first, then re-extract.

## `CHANGELOG.md` per product-protocol

One `CHANGELOG.md` per `<manufacturer>/<product>/<protocol>/` folder. A dual-protocol product (e.g. DDB08 exposed on both ACP2 and Ember+) gets two independent changelogs because each protocol's DM evolves on its own cadence.

Generated on new version addition by `acp diff <existing-version> <new-version>` (#36.c).

Format:

```markdown
# <product> on <protocol> — DM changelog

## [2.4] — 2026-05-15

### Breaking
- Parameter `BOARD.ACP Trace` access changed RW → R
- Enum `Modes` removed value "Experimental" at index 3

### Changed
- Parameter `BOARD.Gain` max: +12 → +18

### Added
- Parameter `BOARD.New Feature` (RW, int, 0-100)

### Removed
- Parameter `STATUS.Legacy Counter`

## [2.3] — 2026-04-19
Initial capture.
```

Section keys (`Breaking`, `Changed`, `Added`, `Removed`) follow [Keep a Changelog](https://keepachangelog.com/). Breaking goes first so it's visible on git blame.

Ordering: newest version at top. Initial capture at bottom with "Initial capture" placeholder.

## Rules

1. **Fixtures under `products/` are generated, never hand-edited.** If a file is wrong, fix the generator and regenerate.
2. **One product per `<manufacturer>/<product>/` folder.** Different vendor products in different folders even if similar.
3. **One protocol per `<manufacturer>/<product>/<protocol>/` folder.** Dual-protocol products have two sibling protocol folders under the same product.
4. **Version folders are immutable.** New firmware = new folder. Never overwrite an existing version.
5. **`meta.json` + `tree.json` are committed as regular blobs (not LFS).** LFS is frozen — keep files < 100 KB or store the capture locally under `bin/devices/` instead.
6. **`wire.jsonl` is committed only if < 100 KB.** Larger wire captures stay local under `bin/devices/captures/` and are referenced in the meta.json `notes` field.
7. **One `CHANGELOG.md` per `<product>/<protocol>/` folder.** Never per version.
8. **When a version folder is added or removed, update the product's `CHANGELOG.md`** — if you're doing it by hand (before #36.c lands), use the format above exactly.

## Generating the layout

| Source | Command | Generates |
|---|---|---|
| Live device | `acp extract <host> --protocol <p> --out tests/fixtures/products/<manufacturer>/<product>/<p>/<version>/` | Full triple (meta + wire + tree) — shipping as **#36.b** |
| Existing `wire.jsonl` | Replay through plugin decoder + re-emit canonical | `tree.json` (regenerate-only), `meta.json` fields refreshed |
| Existing `tree.json` snapshot | `acp convert --in tree.json --out <any-other-format>` | Format conversions — already supported today |
| Diff between two versions | `acp diff <v1>/tree.json <v2>/tree.json` | `CHANGELOG.md` entry — shipping as **#36.c** |

Until `acp extract` (#36.b) lands, fixtures are populated manually by the engineer who captured the device (one-time run of `acp walk --capture <tmpdir>` + hand-stamping `meta.json`).

## Fingerprint example

For a tree.json with 214 objects, canonical encoding:

```
$ openssl dgst -sha256 tree.json
SHA256(tree.json)= 3fa1c4e8...

→ meta.json:
    "dm_fingerprint": "sha256:3fa1c4e8..."
```

Regression catch: if tests/unit/<proto>/replay_test.go decodes the wire.jsonl and produces a tree whose fingerprint differs from the committed `meta.json` fingerprint, the test fails — the decoder moved. Fix the decoder, then regenerate.

## Relationship to existing fixtures

- `tests/fixtures/acp1/slot0_walk.json`, `tests/fixtures/acp2/slot*.json`, `tests/fixtures/emberplus/<port>/` — legacy fixtures, stay in place, tests keep using them.
- `tests/fixtures/products/` — new generated layout, populated by tooling as it ships.
- Eventual full migration is a separate task once `acp extract` and `acp diff` are both live.
