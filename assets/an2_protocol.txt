1 Introduction

This document describes protocol version 1 of the Axonnet2 protocol.

1.1 Definitions

Endpoint             An entry on a Axonnet2 / AxonHSBus identified by a slot nr.
                     EndPoints can appears/disappear dynamically. Note: if ACP is the
Device               protocol, this is a card.
Client               A device running an Axonnet2 server
Slot                 Any TCP client connecting to an Axonnet2 port
Direct connection    Id used to identify an Endpoint, starting from 0
Indirect connection  The Endpoint is on the same device as the Axonnet2 server
                     The Endpoint is on different device than the Axonnet2 server (eg. via
Message
Payload                the Rack Controller (RC))

                     Complete set of information sent from client server or server client
                     (eg. Not packet/frame/etc)
                     All data after an Axonnet2 header.

1.2 Requirements

1.2.1 Goals

-  Axonnet2 should use TCP/IP as base protocol. The Axonnet2 device is the
   server, clients connect to the server.

- An Axonnet2 device should support multiple client connections.

-  Axonnet2 should handle communications from clients to/from slots. Not
   between slots.

-  A device should support a maximum number of connections. This maximum
   is set in the library config (eg 10, 20).

-  After MAX client connections has been reached, new connections should be
   denied.

  - On a protocol error, the connection should be closed.

- Should support 2-way traffic (client server and server client).

-  Should support message payloads up to 4096 bytes (same as HSBus
   payloads).

- Should use TCP port 2072 as server port (Axonnet1 is 2071).

- Should use an internal version (via proto0). This allows protocol changes.

- Should do discovery (over UDP).

                                   2
1.2.2 Slots

- Should support 255 slots, starting from slot 0.

-  Slot 255 is a special broadcast slot (message sent to all Endpoints that are
   online).

- Should signal all clients when an endpoint appears/disappears.

-  The number of slots does not change throughout the device lifetime. The
   EndPoints can appear/disappear dynamically.

-  A device should return the actual number of slots (so an RC could return
   5(=4+1),9(=8+1) or 19(=18+1), since slot0 is also counted.

- Should return an error message when a client uses a non-existing slot.

-  Should return an error message when a client uses an endpoint that is not
   online.

-  Clients should be able to query the total number of slots (of device) and the
   status of each slot.

- Endpoint status can be: (see 3.2.6)

1.2.3 Protocol

- Should be protocol agnostic other than knowing it's payload proto (number).

- Should support at least 128 protocols, starting from 1.

   Should support multiple protocols (eg ACP1/ACP2/..) mixed over a single

- Axonnet2 connection. E.g. task to slot 1 over proto 1 and to slot 4 over

   proto 2.

-  If an Endpoint supports a certain protocol, the Axonnet2 layer should just
   forward it to the Endpoint.

-  Transparantly protocol support at RC / Axonnet2 level (eg. no special code
   for protocol X needed).

- Proto 0 is reserved as 'signalling' protocol for internal (Axonnet2) use.

- Should only forward supported protocols to Endpoints.

-  Should return an error message to client doing request when using a
   non-supported protocol.

-  Should return an error message to client doing request when using a protocol
   that's not supported by that Endpoint.

-  Messages for external (non-zero) protocols are stateless, they can be initiated
   by any Client/Endpoint.

- Mtid not necessary for data (only for internal protocol).

2 Transactions

2.1 Request/Reply

The Mtid of a request and the reply on this request need to match. The
reply message can be an error message. The request and reply messages are
Axonnet2 internal, protocol number is always 0, see 3.2.1.

Client                                Server

                 request:

        [proto=0, slot=n, mtid=x]

                   reply:

        [proto=0, slot=n, mtid=x]

        OR

                   error:

        [proto=0, slot=n, mtid=x]

2.2 Event

An event is also an Axonnet2 internal message, protocol number is always 0,
see 3.2.1.

Client                             Server

                  event:

        [proto=0, slot=n, mtid=0]

2.3 Send data

Can be used for transport of higher protocol messages (eg. ACP2). The mtid
is always 0. Unless an error occurs, Axonnet2 will not send a reply.

Client                                    Server

                     data:

        [proto=proto, slot=n, mtid=0]

        if error

                     error:

        [proto=proto, slot=n, mtid=0]

2.4 Receive data

Used for messages of higher protocol messages (eg. ACP2). An2 keeps no
state for payload protocols, so data messages so can occur any time. Note:
An AN2 server will only send payload events if the client has explicitly en-
abled them with the protocol event enable function

Client                                 Server

                     data:

        [proto=proto, slot=n, mtid=0]

3 Message format

Note: All values are in network byte order(MSB).

         Axonnet2 header                                 payload
                                                           data
magic proto slot mtid type                        dlen
                                                        data[dlen]

Magic    0xC635 (always this number)

 0 request
 1 reply

 0 get version

 0 reserved
 1 reserved
 2 slot info event

3.2.5 Axonnet2 error status

0 protocol error           sent when magic, protocol or type is invalid

1 invalid slot             sent when slot is invalid

2 slot not active          sent when sending data to slot that is not in state
                           'present'

3  slot unsupported proto  sent when sending data to slot with protocol
                           that's not supported

3.2.6 Axonnet2 slot status

 0 empty
 1 powerup
 2 present
 3 error
 4 removed
 5 boot

3.3 Request/Reply messages

The Axonnet2 protocol knows the following 3 request functions: get version,
get device information and get slot information. Each described below.

3.3.1 Get version

With this function you can get the version of the Axonnet2 protocol. This
document describes version 1 of the Axonnet2 protocol. If no error occurs
the version will be replied. The mtid of the request and reply will match.
Else you will get an error message with the same MTID and an error status.

                     Axonnet2 header  func(u8)

5 Discovery

In Axonnet2 it is possible to have discovery over UDP. The UDP port is
2072. When discovery is enabled an UDP broadcast message is sent with
a configurable interval (default a period of 2 seconds). The message is sent
to the local network broadcast address, eg 192.168.1.255. In de table below
you can see the data of an UDP broadcast message. It contains the protocol
name(AN2) with the MAC adress and the ip address, with spaces between
them. The string is ASCII encoded and has a null terminator.

            UDP message data
 AN2 00:aa:11:bb:22:cc 192.168.1.10
