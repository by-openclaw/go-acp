1 Introduction

This document describes protocol version 2 of the ACP2 protocol.
The ACP2 protocol can be used for monitoring and managing devices that
are based on an Axon menu with a set of objects.

                                                2
1.1 Requirements

-  A client should be able to get/set properties of a Card without a-priori knowledge
   about the Card (or menu).

- A client should be able to get/set the value of an Object (for value objects).

-  Messages that do not comply to the protocol must be ignored and an error
   message sent back.

-  It should be possible to enumerate all objects of a Card (that are accessible via the
   root object).

- It should be possible to request all properties of an Object.

-  A string type should support strings up to a 256 bytes, configurable in menu.
   Always UTF-8 Encoded.

-  When an object is read-only, only the value cannot be changed. The max event
   rate and event prio may still be changed.

- ACP2 should have a function to request Object info, given an object-id.

-  ACP2 should have a function to get/set a property given an object-id, Index.
   Which objects can be get/set are shown in the Network API table.

   ACP2 should have a function to request the protocol-version (u8, starts at 1). This

-  number specifies which object-types there are, which properties they have and
   which ACP functions can be used. Rationale: this allows changing the ACP2

   protocol transparently.

-  An object of a certain type shall have all properties as indicated in the
   properties-matrix and no other properties.

- The set of properties for a certain type of object depends on the protocol version.

-  Certain properties may be optional (eg. Alarm/Event options). This is shown in
   the properties matrix.

-  ACP2 should be be transportable over AxonNet2 and AxonHSBus (max message
   size).

-  Is stateless with regard to connections/etc: it just handles packets and sends
   replies/events.

- Should handle single request at a time

-  Disabled parts of the menu are not visible to clients (not shown as children,
   options, etc).

- ACP2 has no concept of slot.

- ACP2 has a concept of request/reply and events.

- ACP2 has no concept of multiple-clients.

- Object labels only may contain out of a-z, A-Z, 0-9, ' ' and '-'.

2 Transactions

2.1 Request/Reply

The mtid of a request and the reply on this request need to match. The reply
message can be an error message.

Client                               Server

                 request:

        [type=0, mtid=x, func=n]

                  reply:

        [type=1, mtid=x, func=n]

        OR

                  error:

        [type=3, mtid=x, func=n]

2.2 Event

The mtid and the func are always 0 for an event.

Client                            Server

                  event:

        [type=2, mtid=0, func=0]

3 Constants

3.1 ACP2 error status

0 protocol error sent on invalid type, function, invalid packet

1 invalid obj-id sent on invalid object id

2 invalid idx sent on invalid index id

3 invalid pid sent on invalid property id or if object doesn't have requested property

4 no access  sent when calling set property on read-only object

5 invalid value sent when setting enum/preset to non-option value, etc

4 Message format

Note: All values are in network byte order(MSB).

4.1 Common header

The ACP2 header describes what type of message it is, can be request, reply,
event or error message. The header fields 3 and 4 are multi-functional, their
meaning depends on the type of message. In most cases it's used for the pid
number. But in some cases its padding, and in case of getting the version
the version value field.

type mtid         ACP2 header  pad0/pid/vers          payload
 u8 u8          func/stat             u8                data
                                                      data[ ]
                    u8

Type           Type of the message, see 3.2
Mtid           Message transfer id, mtid in reply matches mtid in request.
               ACP2 only uses the mtid to send a reply and does no other
Func/stat      checks.
Pad0/pid/vers  Function, see 3.3 or status, see 3.1, for an event message
Data           always 0.
               Multi-function, can be padding (pad0), property id(pid) or
               version number
               Depends on kind of message, can be request or reply header

4.2 Message header

There are 2 types of message headers: request header and reply header. Both
headers can contain a property header when property values are involved. It's
possible to have multiple properties in one message.

4.2.1 Request/Reply header
Common header part of all request and reply headers.

                request/reply header

obj-id     idx  prop(x)                                   prop(x+n)
 u32                                                  multiple of 4bytes
           u32  multiple of 4bytes

                7
obj-id      unique object id number
idx         index number of the object when preset child, otherwise 0
prop (x)    property x
prop (x+n)  property x + n

NOTE:
- idx = 0 for get/set property announce rate or event prio.
- idx 0 (=ACTIVE INDEX) may be used on any request, but the reply will
use the actual active index value instead of 0).

4.3 Request/reply

The ACP2 protocol knows the following 4 functions: get version, get object,
get property and set property. Each described below.

4.3.1 Get version

With this function you can get the version of the ACP2 protocol. This doc-
ument describes version 1 of the protocol. If no error occurs the version will
be replied. The mtid of the request and the reply will match. Else you will
get an error message with the same mtid and an error status.

Request

ACP2 header

type mtid func pad0

type request             see 3.2

mtid x

func get protocol version see 3.3

pad0 0                   padding

Reply
            ACP2 header

  type mtid func vers

type reply               see 3.2

mtid x

func get protocol version see 3.3

vers version             version of the protocol

by id. Only the properties that the object contains will be sent, the order
of properties is by property id, see 5.1. The mtid of the request and reply
message will match. If a request is invalid by for example invalid object id,
an error message will be replied with an error status.

mtid x

pad0 0                        padding

obj-id object id

idx      0

prop (0) property 0

prop (1) property 1

prop (..) rest of properties

replied with an error status.

4.3.4 Set property

With the set property function you can set a property value. You can select
a property value by the property id of an object by id and index. The mtid
of the request and reply message will match. If set property request is in-
valid by for example invalid object id or property id, an error message will
be replied with an error status. The pid value of the ACP2 header needs to
match the pid value of the property header. The following propertie fields
are settable: event delay, value, and event priority.

4.3.5 Error

This message will be sent when a request generates an error. See 3.1 for
possible error status. The mtid will match the mtid of the request.

        ACP2 header

type mtid stat pid

type error  see 3.2

mtid x

stat error status see 3.1

pid 0

5.5 Preset depth

When an object is a preset child, than means that the object has an array
of values of size preset depth. In the table 5.4 you see that the following
property fields can have a depth of preset depth: value, default value, the
mininmal and maximale value. For an object with depth x, the value property
is transmitted x times and each time with his own property header. Also the
preset depth property is present then.

5.5.1 Example preset depth

In the next example you see a part of the properties of a number of two
different objects. The First one is not a preset child. The second one is a
preset child, and has a preset depth of 2. The index number(idx) can't be
0 for an object when it is a preset child, because idx 0 is reserved for the
ACTIVE INDEX.

Object u8 number without preset depth:

        idx = 0

values  125

default value 0

minimum 0

maximum 300

Object u8 number preset depth of 2:

        idx = 100 idx = 200

values  125      10

default value 0  100

minimum 0        10

maximum 300      500

Active index of preset is idx 200, also reachable by idx = 0.
